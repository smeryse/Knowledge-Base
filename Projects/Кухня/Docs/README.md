# Продуктовая база

Главная навигация: [[Projects/Кухня/Views|Index]]

Система для быстрых покупок еды внутри Obsidian.

Главная цель: снять муки выбора и превратить чеки, продукты, запасы дома и список покупок в одну связанную базу.

Смотри на `[[Schema]]` как на главный документ по архитектуре.

- `Products/`, `Recipes/`, `Stores/`, `Receipts/`, `Receipt Items/`, `Pantry/`, `Cooking Log/`, `Meal Plans/`, `Shopping List/` = таблицы

## Сущности

### Product

Карточка товара.

Ключевые поля:

- `type: product`
- `title`
- `barcode`
- `category`
- `brand`
- `store`
- `base_unit`
- `typical_pack_size`
- `typical_pack_unit`
- `perishable`
- `default_shelf_life_days`
- `price`
- `image`

### Store

Магазин, маркетплейс или точка покупки.

### Recipe

Карточка рецепта.

Ключевые поля:

- `type: recipe`
- `title`
- `dish_type`
- `servings`
- `total_time_min`
- `source`
- `products`
- `recipe_status`

Логически состав рецепта должен читаться как набор записей `Recipe Ingredients`, даже если сейчас черновое редактирование делается через таблицу `Ингредиенты` внутри заметки.

### Receipt

Один чек целиком.

В корректной модели чек не бывает пустым: у него должна быть минимум одна связанная запись в `Receipt Item`.

### Receipt Item

Одна позиция внутри чека.

Это главная аналитическая сущность: именно по ней можно смотреть историю покупок и цены, если понадобится.

### Pantry Item

Один конкретный домашний запас.

Нужен, чтобы вести остатки и сроки годности по отдельным партиям, а не абстрактно по товару.

### Shopping Item

Пункт списка покупок.

### Cooking Entry

Одна запись о фактическом приготовлении блюда.

Нужна, чтобы фиксировать не просто рецепт как справочник, а конкретный факт готовки и списание остатков из `Pantry`.

Логически списания внутри готовки должны читаться как набор записей `Cooking Consumptions`, а не как прямая many-to-many связь между `Cooking Log` и `Pantry`.

### Meal Plan

Одна запись о плане питания на месяц.

Нужна, чтобы заранее раскидать рецепты по датам готовки и потом уже от этого строить закупки и фактическую готовку.

Логически сам план должен раскладываться на отдельные `Meal Plan Slots`, а не хранить связь с рецептами как many-to-many напрямую.

## Как пользоваться

### Быстро добавить чек

1. Создай новую заметку в `Projects/Кухня/Receipts/`.
2. Применяй шаблон `Templates/Food/Новый чек.md`.
3. Мастер спросит дату, магазин и товары по одному.
4. Если товар не найден, мастер предупредит и создаст новую карточку.
5. Для скоропорта можно сразу завести дату изготовления в домашнем запасе, а срок годности считать от нее.

### Быстро добавить товар по штрихкоду

1. Открой любую временную заметку или отдельную страницу-сканер.
2. Примени шаблон `Templates/Food/Скан товара.md`.
3. Вставь или отсканируй штрихкод в prompt.
4. Система сначала ищет товар в локальной базе.
5. Если локально не найдено, она пробует найти товар в интернете по штрихкоду.
6. Если найдено, можно принять найденное название и создать карточку с предзаполненными полями.
7. Если не найдено, система создаст карточку вручную и привяжет к ней штрихкод.

### Локальная модель для нормализации

1. Настройки лежат в `Projects/Кухня/System/resolver-config.json`.
2. По умолчанию ожидается `LM Studio` OpenAI-compatible endpoint `http://127.0.0.1:1234/v1`.
3. Модель по умолчанию: `qwen2.5-3b-instruct`.
4. Resolver собирает сырые кандидаты из интернета и просит локальную модель привести их к строгому JSON под твою схему товара.
5. Если локальная модель недоступна, система просто берёт первый удачный источник без LLM-нормализации.
6. Поддерживаются два провайдера: `lmstudio` и `ollama`.

### OCR чека

1. На desktop можно применить шаблон `Templates/Food/OCR чек.md`.
2. Укажи путь к фото чека в `assets/`.
3. Скрипт прогонит изображение через `tesseract` и вставит черновой текст в заметку.
4. Это вспомогательный режим: лучше использовать его для черновика, а не как полностью автоматический парсер.

### Ручной режим

Можно создавать заметки через шаблоны сущностей в `Projects/Кухня/Templates/` и потом редактировать их в `Bases`.

### Где какие шаблоны

1. `Templates/Food/` содержит только команды, которые нужно запускать через Templater.
2. `Projects/Кухня/Templates/` содержит проектные шаблоны сущностей и форму данных.
3. `Templates/scripts/` содержит пользовательские JS-скрипты для этих команд.

### Добавить рецепт

1. Создай новую заметку в `Projects/Кухня/Recipes/`.
2. Примени шаблон `Templates/Food/Новый рецепт.md`.
3. Заполни метаданные рецепта и список `products` ссылками на товары.
4. В секции `Ингредиенты` укажи количества, а в `Шаги` сам процесс приготовления.

### Списать продукты по приготовлению

1. Примени шаблон `Templates/Food/Приготовить блюдо.md`.
2. Выбери рецепт из `Recipes/`.
3. Укажи, сколько порций реально приготовлено.
4. Скрипт разберёт таблицу `Ингредиенты`, спишет поддерживаемые позиции из `Pantry` и создаст запись в `Cooking Log/`.
5. Для автоматического списания в рецепте должны быть числовые количества и единицы `г`, `кг`, `мл`, `л` или `шт`.

### Создать план питания на месяц

1. Примени шаблон `Templates/Food/План питания на месяц.md`.
2. Укажи месяц в формате `YYYY-MM`.
3. Выбери рецепты, которые должны входить в ротацию.
4. Укажи дни готовки через запятую, либо оставь пусто, чтобы использовать `четверг` и `воскресенье`.
5. Скрипт создаст заметку в `Meal Plans/` и раскидает выбранные рецепты по слотам месяца.

## Правила модели

- Один чек = одна заметка в `Receipts/`
- Одна строка в чеке = одна заметка в `Receipt Items/`
- Чек без позиций в модели считается невалидным
- Один физический запас дома = одна заметка в `Pantry/`
- Один рецепт = одна заметка в `Recipes/`
- Одно приготовление блюда = одна заметка в `Cooking Log/`
- Один месячный план питания = одна заметка в `Meal Plans/`
- Связь `Recipes <-> Products` должна идти через `Recipe Ingredients`, а не напрямую
- Связь `Meal Plans <-> Recipes` должна идти через `Meal Plan Slots`, а не напрямую
- Связь `Cooking Log <-> Pantry` должна идти через `Cooking Consumptions`, а не напрямую
- Для скоропорта срок лучше хранить на уровне `Pantry Item`
- Цены и магазины храни в чеках и позициях чека, а не в карточке товара
- В карточке товара `price` можно держать как текущую опорную цену для планирования бюджета
